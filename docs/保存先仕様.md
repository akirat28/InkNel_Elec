# InkNel — 保存先（ストレージパス）方式 詳細仕様

このドキュメントは、InkNel デスクトップ版 (Electron, v0.1.10 以降) で採用した **「保存先フォルダ方式」**の詳細仕様です。React Native で iOS 版を実装する際にこの方式を踏襲し、両 OS で **同じフォルダ** を指定すれば自動的に同じデータを参照・編集できるようにします。

旧 `docs/providor.md`（CloudKit / マニフェスト同期方式）は廃止され、本仕様に置き換わります。

## 1. 設計思想と旧仕様との違い

### 1.1 基本コンセプト

ユーザーが OS で選んだ **任意のフォルダ**（iCloud Drive / Dropbox / Google Drive のローカル同期フォルダ、または通常のローカルフォルダ）を **データの正**として、そこへノート / 画像 / 添付ファイルを直接保存する。

- アプリの DB はあくまで **メタデータの索引キャッシュ** (per-device)
- 同一フォルダを別端末で選択すれば、OS の同期クライアントが内容を共有してくれる → 端末間でも同じノートを編集可能
- 認証フローやアカウント管理は不要（OS の同期クライアントに委譲）

### 1.2 旧 `manifest.json` 方式との差分

| 項目 | 旧 (manifest 方式) | **新 (保存先フォルダ方式)** |
|---|---|---|
| 同期メタデータ | `manifest.json` がノート一覧と updated_at を管理 | **manifest 不使用**。disk 上の `<uuid>.md` ファイルそのものを真として扱う |
| プロバイダ選択 | `'icloud' / 'dropbox' / 'gdrive' / 'none'` から選択、各 OS で検出ロジック | **任意のフォルダ**を OS のフォルダ選択ダイアログでユーザーが指定 |
| 書き込み | userData にも書き、ライトスルーでクラウドにもコピー | **直接**保存先フォルダに書き込み（二重書き込みなし） |
| 同期トリガ | 起動時 / 手動 / mutation (write-through) で 3 層 | **手動の「同期」ボタン**でフォルダ ⇄ DB の差分を解決 |
| 衝突解決 | updated_at ベース Last-Write-Wins | (現状) DB と disk のファイル存在チェックのみ。タイムスタンプ衝突解決は将来拡張 |
| Windows / iOS 検出ロジック | OS ごとに iCloud / Dropbox / Drive の検出処理 | **不要**。フォルダピッカーで一律対応 |

### 1.3 メリット

- **シンプル**: マニフェストや同期エンジン不要
- **OS 中立**: iCloud Drive / OneDrive / Synology / SMB ネットワークドライブなど、任意のフォルダで動く
- **共有が容易**: PC で `iCloud Drive/InkNel/` を選び、iPhone でも同じ場所を選ぶだけ
- **編集の透過性**: 別アプリで `.md` を編集しても、次回 InkNel 同期で取り込める

### 1.4 トレードオフ

- 同時書き込みの整合性は OS の同期クライアントに依存（一般的には Last-Write-Wins）
- リアルタイム反映ではなく、**手動同期トリガ**が必要（自動 watch は将来拡張余地）
- DB が per-device のため、同期せずに別端末で開くと「disk にあるが DB に無い」状態を経て import される

## 2. データレイアウト

### 2.1 ディレクトリ構造

ユーザーが選んだ保存先ルート (= `storage.path`、空なら OS の userData) 直下に以下を作る:

```
<storage-root>/
├── notes/
│   ├── <uuid>.md         # ノート本文 (UTF-8, LF 改行)
│   ├── <uuid>.md
│   └── ...
├── images/
│   ├── <sha256>.<ext>    # 画像 (SHA-256 命名で dedupe)
│   └── ...
└── attachments/
    ├── <sha256>.<ext>    # 添付 (PDF/ZIP 等)
    └── ...
```

注意: **DB ファイル (`inknel.db`) は保存先には置かない**。常に各端末の userData / app sandbox に保持する。これは複数端末からの同時 SQLite 書き込みを避けるため。

### 2.2 ファイル名規約

| 種類 | パターン | 検証正規表現 |
|---|---|---|
| ノート本文 | `<uuid>.md` | `/^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}\.md$/i` |
| 画像 | `<sha256>.<ext>` | `/^[a-f0-9]{64}\.[a-z0-9]{2,5}$/` |
| 添付 | `<sha256>.<ext>` | 同上 |

許容拡張子:
- 画像: `png / jpg / jpeg / gif / webp / svg / avif / bmp` （他は `.bin`）
- 添付: `pdf / zip / lzh / lha / 7z` （他は `.bin`）

### 2.3 Markdown 内の参照

```markdown
![alt](images/<sha256>.<ext>)
[ラベル](attachments/<sha256>.<ext>)
```

相対パスで `images/` / `attachments/` を参照。`<storage-root>` からの相対なのでクロスプラットフォームで動く。

### 2.4 ノートファイル形式（YAML front-matter 付き）

各 `<uuid>.md` は **YAML front-matter** をヘッダに持つ。本文の前に区切り `---` で挟まれたメタ情報を埋め込み、別端末や別アプリで開いてもフォルダ階層・タグ・保護フラグが復元できるようにする。

```markdown
---
title: 買い物リスト
folder: work/ideas
tags: [家事, 急ぎ]
protected: false
secret: false
created_at: 1712800000000
updated_at: 1712850000000
---

# 本文…
```

#### フィールド仕様

| キー | 型 | 必須 | 備考 |
|---|---|:-:|---|
| `title` | string | ✓ | ノートのタイトル |
| `folder` | string | ✓ | 仮想フォルダパス（スラッシュ区切り、ルートは空文字 `""`） |
| `tags` | string[] | ✓ | TagBar で設定したタグ。空なら `[]` |
| `protected` | bool | ✓ | 編集時パスワード要求 |
| `secret` | bool | ✓ | 閲覧時もパスワード要求 |
| `created_at` | int | ✓ | epoch ms |
| `updated_at` | int | ✓ | epoch ms |

#### パーサー仕様

軽量パーサーで以下のみサポート（一般 YAML の全機能には対応しない）:
- スカラー: `key: value`（クォート任意。`"..."` / `'...'` 両対応）
- bool: 小文字 `true` / `false`
- 数値: 整数のみ
- インライン配列: `tags: [a, b, c]`
- ブロック配列:
  ```yaml
  tags:
    - a
    - b
  ```

#### 出力時の規則

- スカラーは原則クォートなし
- `:` `#` `[` `]` `{` `}` `,` `'` `"` を含む値や、`true`/`false`/`null`/`yes`/`no` 等は **ダブルクォート**で包む
- 空配列は `tags: []` の形（`tags:` だけだとブロック形式で曖昧になるため）
- front-matter と本文の間に 1 空行

#### 後方互換

- front-matter が **無い** `.md` も読み込める（メタは空、本文全体を body 扱い）
- 取り込み時に front-matter が無ければ、本文先頭の `# 見出し` をタイトルに、それも無ければ「取り込みノート」を既定タイトルに

PC / iOS は同じ書式を使うこと。これにより端末間でフォルダ・タグ・保護まで完全同期される。

## 3. DB スキーマ

各端末ローカルの SQLite DB (`inknel.db`):

### `notes`

| 列 | 型 | 備考 |
|---|---|---|
| id | TEXT PRIMARY KEY | UUID v4 |
| title | TEXT | |
| folder | TEXT | スラッシュ区切り仮想階層（ファイルシステム上はフラット） |
| protected | INTEGER 0/1 | 編集時パスワード要求 |
| secret | INTEGER 0/1 | 閲覧時もパスワード要求 |
| tags | TEXT | JSON 配列文字列 |
| created_at | INTEGER | epoch ms |
| updated_at | INTEGER | epoch ms |

### `folders`

| 列 | 型 |
|---|---|
| path | TEXT PRIMARY KEY |
| created_at | INTEGER |

### `settings`

| 列 | 型 |
|---|---|
| key | TEXT PRIMARY KEY |
| value | TEXT |

`settings` 内の重要キー: `storage.path`（保存先ルートの絶対パス、空なら既定値）

## 4. 保存先パス設定

### 4.1 設定値

- 設定キー: `storage.path`
- 型: 文字列（絶対パス）
- 既定値: 空文字列 = OS の標準 userData (`app.getPath('userData')` 相当)

### 4.2 設定変更フロー

1. 設定モーダル → 「保存先」カテゴリ
2. 「フォルダを選択」ボタン → OS のフォルダ選択ダイアログ
3. 選ばれたパスを `settings.set('storage.path', path)` で保存
4. ストレージモジュールがキャッシュを破棄して新パスで I/O 開始
5. **既存ファイルは自動移動しない** — ユーザーが手動で旧フォルダの `notes/`, `images/`, `attachments/` を新フォルダにコピーする必要がある旨を UI で明示
6. 「既定に戻す」で空文字列に戻すと userData に戻る

### 4.3 パス解決ロジック（共通仕様）

```pseudo
function getStorageRoot():
    custom = settings['storage.path'].trim()
    if custom and exists(custom) and is_directory(custom):
        ensure_dir(custom)
        return custom
    return platform_default_userdata()
```

キャッシュして毎回 SQLite を引かないようにすること。設定変更時にキャッシュを破棄。

## 5. ファイル I/O ルール

### 5.1 ノート本文

- 書き込み: `<storage-root>/notes/<uuid>.md` に UTF-8, LF で書く
- 読み込み: 同パスから読む。存在しなければ空文字列を返す（エラーにしない）
- 削除: ファイルが存在すれば unlink、無ければ no-op

### 5.2 画像 / 添付

1. クライアントから来たバイナリを SHA-256 でハッシュ化
2. 拡張子を allowlist でサニタイズ（不一致は `.bin`）
3. ファイル名 = `<sha256>.<ext>`
4. **既存ファイルチェック**: 同じファイル名が既にあれば書き込みをスキップ（同一内容のため自動 dedupe）
5. 削除時は SHA-256 名で受け取り、パターン検証してから unlink

### 5.3 セキュリティ

- ファイル名は必ず正規表現でバリデーションし、`..`, `/`, `\` を含む不正名を拒否
- フォルダ作成は `mkdirSync(..., { recursive: true })` で冪等に
- ノート ID は UUID v4 のみ受理（パストラバーサル対策）

## 6. 同期アルゴリズム

「同期」ボタンを押した時に走る処理。**保存先フォルダの内容**と **DB の状態**を比較し、差分を双方向に解消する。

### 6.1 入力データ

- `dbNoteIds`: DB の `notes` テーブルから取得した全 ID
- `diskFiles`: `<storage-root>/notes/` から `readdir` で取得した `.md` ファイル名リスト
- `diskNoteIds`: ファイル名から `.md` を除いた ID 一覧（UUID 風のもののみ）

### 6.2 差分算出 (`scan`)

```pseudo
missingOnDisk = dbNoteIds - diskNoteIds   # DB にあって disk に無い
extraOnDisk = diskNoteIds - dbNoteIds     # disk にあって DB に無い (UUID 風のみ)
```

戻り値:
```ts
{
  storageRoot: string,
  dbNoteCount: number,
  diskFileCount: number,
  missingOnDisk: string[],  // ID 配列
  extraOnDisk: string[],    // ID 配列
}
```

### 6.3 同期実行 (`sync`)

```pseudo
saved = 0
imported = 0

# A: DB → disk
for note in db.list_notes():
    file = join(storage_root, 'notes', note.id + '.md')
    if not exists(file):
        body = read_body_from_db_layer(note.id)  # 端末固有 fallback あり
        write_file(file, body)
        saved += 1

# B: disk → DB
for file in readdir(join(storage_root, 'notes')):
    if not file.endswith('.md'): continue
    id = file[:-3]
    if id in db_ids: continue
    if not is_uuid(id): continue   # 任意名は対象外
    body = read_file(join(storage_root, 'notes', file))
    title = extract_title_from_first_heading(body) or '取り込みノート'
    db.insert_note({
        id,
        title,
        folder: '',
        protected: false,
        secret: false,
        tags: [],
        created_at: now,
        updated_at: now,
    })
    imported += 1

return { saved, imported }
```

### 6.4 タイトル抽出ロジック

```pseudo
function extract_title_from_first_heading(body):
    match = body.match(/^#+\s+(.+)$/m)  # 先頭の見出し行
    return match ? match[1].trim() : null
```

第 1 見出しが無い場合は既定タイトル `'取り込みノート'`。ユーザーは後でリネーム可能。

### 6.5 「ファイルが両方にある」場合

- 現状: **何もしない**（DB の updated_at と disk の mtime のいずれが新しいか比較する処理は将来拡張）
- 端末 A で編集 → 自動保存で disk の `<uuid>.md` が更新される → 端末 B で「同期」を押しても、両側に存在するため何も起きない
- 端末 B で内容を反映するには **明示的な再読み込み**（タブを閉じて開き直すなど）が必要。この点は将来 mtime ベースの比較で改善する予定

### 6.6 メディアファイル（画像 / 添付）

ノート本体と同じ仕組み。基本的には **本文書き出し時に同名ファイルが既に存在すれば skip** で自動 dedupe されるため、明示的な scan/sync は現状省略している。将来的に `images/` / `attachments/` も対象に拡張する余地あり。

## 8. API / IPC 契約

デスクトップ版での実装。iOS 版では関数呼び出しに置き換え可能。

### 8.1 設定操作

- `settings.getAll(): Record<string, string>`
- `settings.set(key: string, value: string): void`
  - `key === 'storage.path'` の場合は同時にストレージルートのキャッシュを破棄

### 8.2 ストレージ操作

- `storage.getRoot(): Promise<string>` — 現在解決済みの実際のルートパス
- `storage.chooseFolder(): Promise<string | null>` — フォルダ選択ダイアログ、キャンセル時 null
- `storage.scan(): Promise<ScanResult>` — DB と disk の差分を返す（[6.2 参照](#62-差分算出-scan)）
- `storage.sync(): Promise<{ saved: number; imported: number }>` — 同期実行（[6.3 参照](#63-同期実行-sync)）

### 8.3 ノート / メディア操作

これらは保存先パスの変更に透過的に対応する（`getStorageRoot()` 経由で内部解決）:

- `notes.create / read-body / update-body / update-meta / delete`
- `images.save / exists`
- `attachments.save / exists / open`

### 8.4 イベント

- `'inknel:notes-changed'` (renderer 内 DOM CustomEvent) — 同期で DB が変わった時に発火、UI 側でノート一覧を再取得

## 9. iOS 版での実装方針

### 9.1 推奨ライブラリ (Expo / React Native)

| 用途 | ライブラリ |
|---|---|
| ファイル I/O | `expo-file-system` |
| SQLite | `expo-sqlite` |
| SHA-256 | `expo-crypto` |
| フォルダ選択 | `expo-document-picker` (`getDocumentAsync({ type: 'public.folder' })` 相当) または **iOS の Files アプリ統合（Document Browser）** |
| iCloud / 外部ストレージ | `UIDocumentPickerViewController` のラッパー（カスタムネイティブモジュール）または `expo-file-system` の `StorageAccessFramework` 風機能 |

### 9.2 Security-Scoped Bookmarks（重要）

iOS / iPadOS では、ユーザーがピッカーで選んだフォルダへのアクセス権はアプリ実行ごとに失われる。**Security-Scoped Bookmark** を保存し、毎起動時に解決する必要がある:

```swift
// Swift 側ネイティブモジュールでの実装イメージ
let bookmarkData = try url.bookmarkData(
    options: [.minimalBookmark],
    includingResourceValuesForKeys: nil,
    relativeTo: nil
)
// bookmarkData を Base64 で TS 側に渡し、settings に保存

// 起動時に復元
var stale = false
let url = try URL(
    resolvingBookmarkData: data,
    options: [],
    relativeTo: nil,
    bookmarkDataIsStale: &stale
)
url.startAccessingSecurityScopedResource()
defer { url.stopAccessingSecurityScopedResource() }
// この URL を保存先ルートとして使う
```

### 9.3 storage.path の保存形式

iOS では絶対パスではなく **Base64 エンコードされた bookmark data** を `storage.path` に保存することを推奨:

```ts
// settings.storage.path
// macOS / Linux / Windows: 絶対パス文字列 ("/Users/me/...")
// iOS / iPadOS:            Base64 エンコードされた security-scoped bookmark
```

ストレージ層は OS を判別して使い分ける:

```pseudo
function getStorageRootIOS():
    bookmark_b64 = settings['storage.path']
    if not bookmark_b64: return getDocumentDirectory()  # アプリ sandbox
    url = resolve_bookmark(bookmark_b64)
    if not url: return getDocumentDirectory()
    return url.path  # security-scoped で開始済み
```

### 9.4 iCloud Drive 連携

ユーザーが iCloud Drive 内のフォルダを選ぶと、OS が自動的に他端末に同期する。アプリは Files アプリの DocumentPicker を経由してこのフォルダにアクセスできる。

**推奨セットアップ手順（ユーザー向け）**:
1. PC アプリで「設定 → 保存先 → フォルダを選択」
2. iCloud Drive 内に新規フォルダ作成（例: `iCloud Drive/InkNel/`）
3. 同じ Apple ID でログインした iPhone / iPad で InkNel iOS 版を開く
4. 「設定 → 保存先 → フォルダを選択」で Files アプリ → iCloud Drive → InkNel を選ぶ
5. 「同期」ボタンで初回取り込み → 以後は両端末で同じデータを編集可能

### 9.5 バックグラウンド処理

iOS のバックグラウンド実行は厳しく制限される。

- **フォアグラウンド復帰時**: `AppState.addEventListener('change', ...)` で active 復帰を検知し、自動で `storage.scan()` を走らせて差分があるか UI に通知
- **明示的な「同期」ボタン**: 確実な反映はユーザー操作で行う
- **iCloud のファイル変更通知**: `NSMetadataQuery` を NSFilePresenter 風に組めば自動検知も可能（実装コスト高、初版では省略推奨）

## 10. クロスプラットフォーム互換性

### 10.1 共有時の必須条件

両 OS で **同じ物理フォルダ**（iCloud Drive 等の同期実体）を選ぶこと。

### 10.2 ファイル形式の共通項

- **改行コード**: `\n` (LF) で統一
- **エンコーディング**: UTF-8 (BOM なし)
- **タイムスタンプ**: SQLite `notes.updated_at` は epoch ms (int)。disk 側の mtime はファイルシステム任意

### 10.3 ID の不変性

- 一度生成した UUID は **変更しない**
- 端末 A で `<uuid>.md` を作る → 同じ UUID で disk に存在 → 端末 B が import する時は同じ UUID で DB 登録
- 別 UUID で生成すると重複扱いになるので、import の際は **必ず disk のファイル名を ID として採用** すること

### 10.4 メタ情報の取り扱い（front-matter による完全同期）

各 `.md` ファイルの先頭に **YAML front-matter** を埋め込むことで、`folder` / `tags` / `protected` / `secret` / `created_at` / `updated_at` も端末間で共有される（[§2.4 参照](#24-ノートファイル形式yaml-front-matter-付き)）。

書き出しのライフサイクル:
- ノート作成 / 本文編集 / メタ変更 / 保護トグル のいずれかが起きるたびに、`<uuid>.md` を **front-matter + 本文** で上書き
- DB の `notes` テーブルが正規メタ、front-matter はそれをそのまま反映するミラー

取り込みのライフサイクル:
- 別端末でディスク上の `.md` を見つけたら front-matter をパース
- メタ情報を DB の `notes` レコードに復元（`folder`, `tags`, `protected`, `secret`, タイムスタンプ含む）
- front-matter が無いファイル（手書き、別アプリ作成など）は、ルートフォルダ・空タグ等の既定値で取り込み（後方互換）

### 10.5 メディア（画像・添付）

- SHA-256 命名なので衝突しない
- 端末 A で `images/abc.png` を保存 → 同じパスに OS が同期 → 端末 B でも `images/abc.png` として参照される
- import / scan は本文と同じくファイルの存在チェックで dedupe される

## 11. エッジケース

### 11.1 保存先を変更したが既存ファイルが残る

- 旧 storage.path のファイルは自動的には移動されない
- UI で**警告メッセージ**を出し、ユーザーが手動コピーするよう案内する
- 移動後は旧フォルダから DB のメタは引き続き参照されるが、disk に該当 `.md` が無いため、再同期で「missingOnDisk」扱いになり書き出される（実質的にコピー成立）

### 11.2 disk 上に .md があるが UUID 形式でない

- `extraOnDisk` の対象外。**自動 import しない**
- ユーザーが「ファイルの読み込み」メニュー（既存機能）で明示的に取り込めば、新しい UUID が割り当てられて取り込まれる

### 11.3 disk 上の .md と DB が同じ ID で内容が違う

- 現仕様では検知しない（未対応）
- 端末 A で disk 編集 → 端末 B が同期 → 何もしない（両方に存在するため）→ 端末 B のアプリで開くと **disk 側の最新内容が表示される**（`read-body` は常に disk から読むため）
- 端末 B で編集すると disk に上書きされる → 端末 A の DB に updated_at が古いまま残る

将来 mtime / hash ベースの比較で改善予定。

### 11.4 同時編集

- OS の同期クライアントが Last-Write-Wins で解決
- 編集が衝突した場合、どちらかの内容が失われる可能性
- iCloud Drive は内部的に競合バージョンを残すこともあるが、アプリ側では特別な処理なし

### 11.5 保存先フォルダが消失 / アクセス不能

- `getStorageRoot()` が例外を吐かないように、`existsSync` 失敗時は userData にフォールバック
- UI で「保存先が見つかりません」エラーを出す処理は将来追加（現状はサイレント fallback）

### 11.6 大量ノート

- `storage:scan` は `readdir` 1 回 + Set 操作だけなので O(N)
- 数千件まで実用的、数万件で readdir が遅くなる可能性あり（将来的にバッチ化）

## 12. テスト観点

### 12.1 単体

- `getStorageRoot()` が空・既定・カスタム値で正しい値を返す
- 不正なパス（存在しない / ファイル）の時に userData にフォールバックする
- ファイル名のバリデーション（UUID, SHA-256 パターン）が機能する

### 12.2 統合

- 保存先設定変更後にノート作成 → 新しいフォルダに `.md` が生成される
- 既存設定下でノート編集 → 既存パスに上書きされる
- `storage:scan` が DB と disk の差分を正しく算出する
- `storage:sync` が `missingOnDisk` を全件書き出し、`extraOnDisk` を全件取り込む

### 12.3 シナリオ（クロスプラットフォーム）

- 端末 A で 5 ノート作成 → iCloud に同期 → 端末 B でフォルダ選択 + 同期 → 5 ノート取り込まれる
- 端末 A でノート編集 → 端末 B でアプリ再起動 → タブを開き直すと最新内容
- 端末 A でノート削除 → 端末 B では DB に残る（手動削除が必要、将来課題）

### 12.4 セキュリティ

- 不正な ID（`../etc/passwd`, バックスラッシュ等）でファイル操作しようとしてもブロックされる
- 保存先が `/` や `~` などのシステムディレクトリでも、`<root>/notes/` 配下でしか書き込みしない
- iOS では security-scoped bookmark の有効性チェックが起動時に走る
